## 제품 요약

- 제품명: 라멘 도장깨기
- 출시 방향: 웹사이트 + 토스 미니앱
- 앱 유형: 비게임
- 핵심 가치: 다녀온 라멘집과 먹은 메뉴를 기록하고, 국물/면/토핑/종합 평점과 메모를 쌓아 나만의 라멘 취향 데이터를 만든다.
- MVP 방향: 로그인 없이 시작한다. 공용 라멘집 catalog는 서버에서 받고, 개인 방문 기록과 위시리스트는 localStorage에 저장한다.
- 수집 감각: 전체 라멘집을 모두 채우는 진행률이 아니라, 최대치 없는 누적 도장/기록 감각을 준다.

## 디자인 목표

- 토스 미니앱 안에서 어색하지 않은 모바일 앱 화면을 우선한다.
- Toss 앱 안에서 어색하지 않은 neutral surface, 명확한 primary action, 간결한 리스트 구조를 쓴다.
- 라멘 고유의 개성은 썸네일, 도장/기록 카피, 취향 요약에만 작게 넣는다.
- 크림/브라운/오렌지 계열로만 흐르는 라멘집 감성 UI는 피한다.
- 랜딩페이지 hero나 홍보 문구보다, 반복 사용 가능한 기록 도구의 밀도와 명확성을 우선한다.

## 현재 코드 기준 디자인 앵커

최신 구현은 이미 토스스러운 방향을 따르고 있다. Stitch 시안도 이 방향을 강화해야 한다.

- 일반 웹 배포도 같이 해야 하므로 `@toss/tds-mobile` 런타임 package는 사용하지 않는다.
- 현재 primary color는 shoyu/amber 계열이다. 화면 전체를 갈색/주황색으로 덮지 않고 CTA, 선택 상태, 작은 강조에만 제한적으로 쓴다.
- 기본 텍스트/배경/경계 색은 Toss neutral 계열에 가깝다.
  - text: `#191f28`
  - secondary text: `#6b7684`, `#8b95a1`
  - background: `#f9fafb`, `#f2f4f6`
  - border: `#e5e8eb`, `#d1d6db`
- Button, TextField, TextArea, Badge는 native HTML 기반 local component를 사용한다.
- 앱인토스에서는 자체 상단 topbar가 네이티브 내비게이션 바와 중복되지 않아야 하고, 하단 탭바는 토스 가이드의 플로팅 형태를 따라야 한다.
- 라멘 이미지는 장식 배경이 아니라 라멘집 카드의 thumbnail/placeholder 용도로만 쓴다.
- 시안은 현재 코드보다 더 “토스 앱 안의 기능 화면”처럼 보여야 하며, 라멘집 브랜딩 사이트처럼 보이면 안 된다.

현재 구현에서 유지할 방향:

- 웹과 앱인토스가 같은 `apps/web` 화면을 공유한다.
- 홈은 요약, 취향 통계, 최근 방문 기록을 먼저 보여준다.
- 라멘집 catalog는 서버 API에서 오고, 방문 기록과 위시리스트는 로컬 저장소에 둔다.
- 방문 추가는 “라멘집 검색/선택 → 리뷰 작성”의 2단계 흐름이다.
- 방문 리뷰는 라멘 특화 평가 축인 국물/면/토핑/종합 별점을 핵심 입력으로 둔다.

## 공통 UI 요구사항

- 모바일은 360px 폭에서도 깨지지 않아야 한다. 시안 대표 폭은 390px로 잡되, 360px을 최소 검수 폭으로 본다.
- 상단에는 앱 이름과 핵심 탭을 둔다: 홈, 라멘집, 방문 추가, About.
- 주요 CTA는 `방문 추가`다.
- 버튼, 입력, 배지, 리스트 행은 일반 웹에서도 깨지지 않는 native HTML 기반으로 만들되 앱인토스 UI/UX 가이드에 맞는 여백과 형태를 우선한다.
- 카드 안에 또 카드를 넣지 않는다.
- 별점은 라멘 전용 평가 축 4개를 명확히 보여준다: 국물, 면, 토핑, 종합.
- 기본 썸네일이 필요한 라멘집 카드는 placeholder가 어색하지 않아야 한다.
- 빈 상태, 로딩, 오류 상태도 시안에 포함한다.

## 정보 구조

```text
/
  홈: 요약, 취향 통계, 최근 방문 기록

/shops
  라멘집 목록: 서버 catalog 검색, 필터, 방문 여부 확인

/shops/:shopId
  라멘집 상세: 라멘집 정보, 가고싶음, 방문 기록 추가, 방문 기록 목록

/visits/new
  방문 추가: 라멘집 검색/선택, 리뷰 폼 작성

/visits/:visitId
  방문 기록 상세/수정: 방문 기록 수정, 삭제

/about
  앱 컨셉 설명
```

## 화면별 요구사항

### 1. 홈

목적:

- 사용자가 앱을 열자마자 지금까지 쌓인 라멘 기록을 본다.
- 새 방문 기록을 남기거나 라멘집 목록으로 이동한다.

필수 요소:

- 상단 요약 영역
  - 카피: `다녀온 맛을 쌓고, 다음 한 그릇을 고릅니다.`
  - 지표: 전체 라멘집 수, 방문 수, 기록 수
- 내 취향 요약
  - 평균 만족도
  - 재방문 의향 비율
  - 자주 먹은 태그
  - 자주 간 곳
- 방문한 라멘집
  - 최근 방문 기록 최대 5개
  - 메뉴명, 라멘집 이름, 방문일, 종합 평점, 메모 일부
- CTA
  - `방문 추가`
  - `라멘집 보기`

상태:

- 방문 기록이 없으면 `방문 기록을 남기면 내 취향 요약이 생깁니다.`
- 최근 방문 기록이 없으면 `아직 방문 기록이 없습니다.`
- API 오류가 있으면 `API 서버 연결을 확인해주세요.`

디자인 메모:

- 진행률 bar나 전체 달성률은 쓰지 않는다. 라멘집 전체 개수의 max가 없기 때문이다.
- 통계는 과한 chart보다 숫자와 짧은 label 중심으로 둔다.

### 2. 라멘집 목록

목적:

- 서버에 저장된 라멘집 catalog를 검색하고, 방문 여부와 태그를 빠르게 훑는다.
- 방문 기록을 남길 라멘집을 찾는다.

필수 요소:

- 제목: `라멘집`
- 설명: `다녀온 곳과 가볼 곳을 한 목록에서 관리합니다.`
- 필터
  - 이름 검색
  - 태그 검색
  - 방문 여부: 전체, 방문함, 미방문
- 라멘집 카드
  - 썸네일
  - 라멘집 이름
  - 주소
  - 태그 배지
  - 방문 상태 배지: 방문함, 미방문
  - 평균 평점
  - 상세 이동 버튼

상태:

- 검색 결과 없음: `조건에 맞는 라멘집이 없습니다.`
- 오류: `목록을 불러오지 못했습니다.`

디자인 메모:

- 카드형 목록이지만 너무 장식적이지 않게, 토스의 리스트/셀 구조에 가깝게 만든다.
- 썸네일이 없어도 기본 라멘 이미지 placeholder가 자연스럽게 보여야 한다.

### 3. 방문 추가

목적:

- 사용자가 DB에 저장된 라멘집을 검색해 선택하고, 방문 리뷰를 남긴다.

필수 요소:

- 제목: `방문 추가`
- 설명: `저장된 라멘집을 검색해서 고른 뒤, 이번 방문의 맛을 남깁니다.`
- 라멘집 검색
  - 검색 input
  - 검색 결과 리스트
  - 선택된 라멘집 강조
- 리뷰 작성 폼
  - 방문일
  - 메뉴
  - 국물 별점 1~5
  - 면 별점 1~5
  - 토핑 별점 1~5
  - 종합 별점 1~5
  - 재방문 의사: 있음, 없음
  - 메모
  - 등록 버튼
- 보조 CTA
  - `라멘집 목록에서 찾기`

상태:

- 라멘집 미선택: `리뷰를 작성할 라멘집을 먼저 선택해주세요.`
- 검색 결과 없음: `검색된 라멘집이 없습니다.`
- 등록 실패: `방문 기록을 등록하지 못했습니다.`

디자인 메모:

- 라멘집 선택 전과 선택 후의 단계가 분명해야 한다.
- 별점은 조작하기 쉬운 크기여야 한다.
- 리뷰 텍스트는 긴 글보다 메모장 감각으로 가볍게 작성할 수 있어야 한다.

### 4. 라멘집 상세

목적:

- 한 라멘집의 기본 정보와 내 방문 기록을 보고, 바로 기록을 추가한다.

필수 요소:

- 뒤로가기: `목록`
- 라멘집 이름
- 주소
- 가고싶음 토글
  - 미등록: `가고싶음`
  - 등록됨: `가고싶음 해제`
- 방문 기록 추가 폼
  - 방문 추가 화면과 같은 리뷰 필드
  - 라멘집은 현재 라멘집으로 고정
- 방문 기록 목록
  - 메뉴명
  - 방문일
  - 종합 평점
  - 메모 일부

상태:

- 로딩: `불러오는 중입니다.`
- 라멘집 없음: `라멘집을 찾을 수 없습니다.`
- 방문 기록 없음: `아직 기록이 없습니다.`

디자인 메모:

- 상세 화면은 정보 페이지라기보다 “이 라멘집에 또 기록하기” 화면이다.
- 가고싶음은 과한 찜 UX보다 조용한 토글 액션이면 충분하다.

### 5. 방문 기록 상세/수정

목적:

- 이미 작성한 방문 기록을 확인하고 수정하거나 삭제한다.

필수 요소:

- 뒤로가기: `라멘집`
- 제목: 메뉴명
- 보조 정보: 라멘집 이름, 방문일
- 삭제 버튼
- 방문 기록 수정 폼
  - 라멘집
  - 방문일
  - 메뉴
  - 국물/면/토핑/종합 별점
  - 재방문 의사 있음/없음
  - 메모
  - 수정 버튼

상태:

- 로딩: `불러오는 중입니다.`
- 기록 없음: `방문 기록을 찾을 수 없습니다.`

디자인 메모:

- 삭제는 위험 액션으로 구분한다.
- 수정 화면은 새 방문 추가와 같은 컴포넌트 언어를 유지한다.

### 6. About

목적:

- 서비스 컨셉을 짧게 설명한다.

필수 요소:

- 제목: `라멘집 도장깨기를 위한 지도형 기록장입니다.`
- 설명: `다녀온 곳과 맛의 기억을 남기고, 언젠가 다시 가고 싶은 라멘집을 모아둡니다. 하나씩 찾아가며 나만의 라멘 지도를 채워보세요.`

디자인 메모:

- About은 마케팅 랜딩페이지가 아니다.
- 앱 안의 짧은 안내 화면 정도로 작게 둔다.

## Stitch 요청 프롬프트 초안

```text
라멘집 방문 기록 앱 "라멘 도장깨기"의 모바일 우선 UI 시안을 만들어줘.

출시 타겟은 웹사이트와 토스 미니앱이고, 토스 앱 안에서 자연스럽게 보이는 비게임 서비스여야 해. 토스다운 neutral surface, 명확한 primary action, 간결한 리스트 구조를 우선해줘.

현재 코드는 일반 웹 배포도 해야 해서 앱인토스 전용 TDS package에는 의존하지 않아. 대신 토스다운 여백/타이포/리스트 감각은 유지하고, brand primary는 shoyu/amber 계열로 제안해줘. 단, 화면 전체를 갈색/주황색으로 덮지 말고 neutral UI 위에서 주요 CTA, 선택 상태, 작은 강조에만 제한적으로 써줘. 라멘 고유의 시각 요소도 썸네일, 기록/도장 카피, 취향 요약 정도에만 작게 사용해줘.

피해야 할 것: 크림/브라운/오렌지 계열로만 흐르는 과한 라멘집 감성 UI, 마케팅 랜딩페이지, 큰 hero 페이지, 음식점 브랜드 사이트처럼 보이는 화면. 만들어야 할 것은 실제 앱인토스 안에서 바로 쓸 수 있는 기록형 기능 화면이야.

MVP는 로그인 없이 시작해. 공용 라멘집 catalog는 서버에서 받아오고, 개인 방문 기록과 위시리스트는 로컬에 저장하는 구조야. 전체 라멘집을 모두 방문하는 목표가 아니므로 진행률 max는 없어. 대신 방문 기록이 계속 쌓이는 도장/수집 감각과 내 취향 데이터가 쌓이는 느낌을 줘.
```
